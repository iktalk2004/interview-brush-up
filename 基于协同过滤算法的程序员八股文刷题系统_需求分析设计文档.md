## 1. 项目概述与背景

### 1.1 项目背景

在当前互联网行业求职竞争日益激烈的背景下，程序员技术面试中的"八股文"（即计算机基础知识、编程语言特性、框架原理等高频面试题）已成为求职者必须掌握的核心内容。然而，现有的刷题平台普遍存在以下问题：

- 题目推荐缺乏个性化，无法根据用户的知识薄弱点和学习偏好进行精准推送
- 简答类面试题缺少有效的自动评估机制，用户难以判断自身作答质量
- 缺少针对"八股文"这一特定场景的垂直化刷题产品

### 1.2 项目简介

本系统是一款面向程序员群体的八股文面试题在线刷题平台，支持简答题和代码题两种题型。系统的核心亮点在于引入多种推荐算法（基于用户的协同过滤、基于物品的协同过滤、基于内容的推荐），根据用户画像、刷题行为和偏好数据实现个性化题目推荐，并提供算法效果对比分析功能。同时，系统集成了基于语义相似度模型和DeepSeek API的智能评分功能，为简答题提供自动化评估。

### 1.3 技术栈

| 层次 | 技术选型 |
|------|----------|
| 前端 | Vue.js |
| 后端 | Django (Python) |
| 数据库 | SQLite（初期），后续迁移至MySQL |
| 在线判题 | Judge0（开源Online Judge系统） |
| AI评分 | 语义相似度模型（sentence-transformers等多模型综合） + DeepSeek API |
| 推荐算法 | User-based CF / Item-based CF / Content-based推荐 |

## 2. 系统目标与范围

### 2.1 系统目标

1. 构建一个覆盖主流程序员面试岗位知识点的八股文题库系统
2. 实现基于多种协同过滤算法的个性化题目推荐，并支持算法效果对比
3. 提供简答题AI智能评分和代码题在线自动判题功能
4. 提供完善的刷题统计、错题回顾、排行榜等辅助学习功能

### 2.2 系统范围

**包含在范围内：**

- 用户注册/登录/找回密码/个人信息管理
- 题库浏览、搜索、分类筛选
- 简答题作答与AI评分（内嵌模型/DeepSeek API可选，评分归一化到0-100分）
- 代码题在线编写、提交与自动判题（Python、Java、C++），采用异步判题
- 个性化题目推荐（多种算法）及管理员端算法效果对比
- 每日推荐（每天5道，定时生成）
- 专题练习、错题回顾
- 个人数据看板（含页面停留时长统计）与排行榜
- 题目评论区（支持嵌套回复）
- 题目收藏
- 系统公告管理
- 管理员后台（题目/用户/分类/公告管理、数据统计、推荐算法对比）

**不包含在范围内：**

- 模拟面试组卷功能
- 移动端适配
- UI风格切换（暗黑模式等）
- 隐藏测试用例机制
- 提交频率限制与防作弊机制（后续迭代）

## 3. 用户角色分析

### 3.1 普通用户（刷题者）

**用户画像：** 正在准备程序员技术面试的求职者，包括应届毕业生、社招跳槽者等，覆盖前端、后端、算法、测试等各方向。

**核心诉求：**

- 高效地刷到与自身求职方向匹配的高频面试题
- 了解自己的知识薄弱点，获得针对性推荐
- 对简答题的作答质量获得即时反馈
- 通过代码题练习提升编程能力
- 追踪学习进度，保持学习动力

**权限：**

- 注册、登录、修改个人信息、邮箱验证码找回密码
- 浏览题库、搜索题目、按分类筛选
- 作答简答题和代码题，支持重复练习
- 选择评分方式（内嵌模型/DeepSeek API）
- 查看每日推荐、专题练习、错题回顾
- 查看个人数据看板
- 收藏题目
- 发表和回复评论
- 查看排行榜

### 3.2 管理员

**用户画像：** 系统运营维护人员。

**核心诉求：**

- 高效管理题库内容
- 监控系统运行状态和用户使用情况
- 对比不同推荐算法的效果

**权限（在普通用户权限之上）：**

- 题目CRUD（增删改查）及Excel批量导入
- 查看每道题目的提交总次数、平均得分/通过率
- 用户管理（查看列表、禁用账号等）
- 分类管理（岗位方向、知识领域的增删改查）
- 公告管理（发布、编辑、删除系统公告）
- 系统整体数据统计
- 推荐算法效果对比面板（切换算法、查看指标）

## 4. 功能性需求

### 4.1 用户系统模块

#### 4.1.1 用户注册

**功能描述：** 新用户填写信息完成注册。

**注册信息：**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 用户名 | 文本 | 是 | 唯一，用于登录和展示 |
| 邮箱 | 文本 | 是 | 唯一，用于登录和找回密码 |
| 密码 | 文本 | 是 | 加密存储 |
| 头像 | 图片上传 | 否 | 用户自定义上传，有默认头像 |
| GitHub ID | 文本 | 否 | 可选填写 |
| 感兴趣的题目类型 | 多选 | 是 | 从岗位方向/知识领域中选择，用于冷启动推荐 |
| 技术水平自评 | 单选 | 是 | 初级/中级/高级 |
| 学历 | 单选 | 否 | 大专/本科/硕士/博士/其他 |

**业务规则：**

- 用户名和邮箱不可重复
- 密码需满足最低强度要求（不少于8位，包含字母和数字）
- 感兴趣的题目类型至少选择1项
- 注册成功后自动登录

#### 4.1.2 用户登录

**功能描述：** 支持用户名或邮箱 + 密码登录。

**业务规则：**

- 登录失败给出错误提示
- 登录成功后跳转至首页
- 使用JWT进行身份认证

#### 4.1.3 找回密码

**功能描述：** 用户通过邮箱验证码重置密码。

**流程：**

1. 用户输入注册邮箱
2. 系统发送验证码到邮箱
3. 用户输入验证码并设置新密码
4. 密码重置成功

**业务规则：**

- 验证码有效期5分钟
- 验证码为6位数字
- 同一邮箱60秒内不可重复发送

#### 4.1.4 个人信息管理

**功能描述：** 用户可修改个人信息，包括头像、用户名、GitHub ID、兴趣方向、技术水平、学历、密码。

### 4.2 题库系统模块

#### 4.2.1 题目分类体系

**分类维度：**

采用两级分类结构：

- **一级分类（岗位方向）：** 如Python后端、Java后端、前端开发、算法工程师、数据库、计算机网络、操作系统等
- **二级分类（知识领域）：** 如Python后端下设Django、FastAPI、Flask等；数据库下设MySQL、Redis、MongoDB等

**难度等级：** 简单 / 中等 / 困难（三级）

#### 4.2.2 题目数据模型（统一基础表 + 扩展表）

系统采用统一题目基础表 + 类型扩展表的设计，所有题目共享基础信息，不同题型的专有字段存放在各自的扩展表中。收藏、评论、推荐、答题记录等模块统一通过基础表的`question_id`关联，无需额外携带`question_type`字段。

**题目基础表字段：**

| 字段 | 类型 | 必填 | 说明 |

| 题目ID | 自增整数 | 自动生成 | 主键 |
| 题目类型 | 枚举 | 是 | text(简答题) / code(代码题) |
| 标题 | 文本 | 是 | 题目标题 |
| 一级分类 | 外键 | 是 | 关联岗位方向 |
| 二级分类 | 外键 | 是 | 关联知识领域 |
| 难度等级 | 枚举 | 是 | 简单/中等/困难 |
| 创建时间 | 时间戳 | 自动 | — |
| 更新时间 | 时间戳 | 自动 | — |

**简答题扩展表字段：**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| question_id | FK → questions | 是 | 关联基础表，一对一 |
| content | 富文本 | 否 | 题干补充描述 |
| standard_answer | 富文本 | 是 | 标准答案 |
| explanation | 富文本 | 否 | 详细解析 |

**代码题扩展表字段：**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| question_id | FK → questions | 是 | 关联基础表，一对一 |
| description | 富文本 | 是 | 题目描述（题意、输入输出格式、示例等） |
| test_cases | JSON | 是 | 测试用例（输入和期望输出） |
| reference_code | 文本 | 否 | 参考解法 |
| code_template | JSON | 否 | 各语言初始代码模板 |
| time_limit | 整数(ms) | 是 | 代码执行时间上限 |
| memory_limit | 整数(MB) | 是 | 代码执行内存上限 |

#### 4.2.3 题库浏览与搜索

**功能描述：**

- 支持关键词搜索题目标题和描述
- 分页展示，每页显示可配置的数量

#### 4.2.4 题目数据导入

**功能描述：** 管理员通过上传Excel文件批量导入题目。

**业务规则：**

- 支持简答题和代码题分别导入

### 4.3 刷题系统模块

#### 4.3.1 重复练习策略

用户可以对同一道题进行多次重复练习，系统保留全部答题记录，不同模块依据不同取值规则使用数据：

| 模块 | 取值规则 | 理由 |
|------|----------|------|
| 个人看板-正确率 | 取每道题的**最优成绩** | 反映用户当前真实水平 |
| 个人看板-刷题总数 | 按**去重题目数**统计 | 避免重复刷同一题虚增数量 |
| 个人看板-学习时长 | 累加**所有记录**的时长 | 重复练习确实花了时间 |
| 排行榜-刷题数量 | 按**去重题目数** | 公平性 |
| 排行榜-正确率 | 取每道题的**最优成绩**后计算 | 鼓励重复练习直到掌握 |
| 错题本 | 根据**最近一次**提交判定 | 最近一次通过了就移出错题本 |
| 推荐算法 | 可使用**全部记录** | 多次做错权重更高，反映真实薄弱点 |
| 题目统计（管理员） | 提交总次数取**全部记录**，通过率取**每用户最优后的通过率** | 区分"提交热度"和"题目难度" |

系统通过`user_question_status`汇总表维护每个用户在每道题上的聚合状态（提交次数、最高分、最近分数、是否掌握等），每次提交答案时同步更新，各模块查询时直接读取汇总表，避免频繁聚合原始记录。

#### 4.3.2 简答题作答流程

**流程：**

1. 用户进入简答题详情页，查看标题和题干补充
2. 在答题区域输入自己的答案
3. 选择评分方式：内嵌模型评分 或 DeepSeek API评分
4. 提交答案
5. 系统返回评分结果（归一化到0-100分，保留两位小数）及评语
6. 用户可查看标准答案和详细解析
7. 系统记录本次答题数据（答案内容、评分、评分方式、耗时、时间戳）
8. 同步更新`user_question_status`汇总表

**AI评分 — 内嵌模型评分：**

- 采用多种语义相似度模型（如sentence-transformers系列）综合计算用户答案与标准答案的语义相似度
- 多模型分数加权或投票得出最终分数
- 原始相似度分数（0-1）归一化映射至0-100分区间，保留两位小数
- 返回归一化分数和简要评价

**AI评分 — DeepSeek API评分：**

- 将题目（标题 + 题干）和用户答案发送至DeepSeek API
- 由DeepSeek自行分析并返回评分和评语
- 不发送标准答案，完全由AI独立评判
- API返回结果归一化映射至0-100分区间，保留两位小数
- DeepSeek API不可用时提示用户切换为内嵌模型评分

**评分归一化说明：**

两种评分方式的原始分数标准不同（内嵌模型为语义相似度0-1，DeepSeek可能返回不同标准），系统统一将所有评分映射到0-100分区间（保留两位小数），确保错题本判定、排行榜统计、个人看板数据的一致性。答题记录中同时保存评分方式字段以供追溯。

#### 4.3.3 代码题作答流程

**流程：**

1. 用户进入代码题详情页，查看题目描述
2. 选择编程语言（Python / Java / C++）
3. 在代码编辑器中编写代码（可基于代码模板）
4. 点击"运行"执行代码，查看运行结果
5. 点击"提交"进行判题
6. 前端显示"判题中..."状态
7. 后端通过Judge0异步执行判题，前端定时轮询结果
8. 返回判题结果：通过/未通过、各测试用例执行结果、耗时和内存占用
9. 用户可查看参考代码
10. 系统记录本次答题数据并同步更新`user_question_status`汇总表

**Judge0集成方案：**

- 部署Judge0的Docker实例
- 后端通过API调用Judge0，传入源代码、语言、测试输入
- 获取执行结果与标准输出进行比对
- 支持语言：Python 3、Java、C++

**异步判题流程：**

1. 用户提交代码，后端创建判题任务，返回任务ID
2. 后端将任务提交至Judge0执行
3. 前端携带任务ID定时轮询（如每2秒一次）判题状态
4. Judge0执行完成后，后端更新任务状态和结果
5. 前端轮询到已完成状态，展示判题结果
6. 轮询超时（如60秒）提示用户稍后查看

#### 4.3.4 专题练习

**功能描述：** 用户选择一个分类（一级或二级）和可选的难度等级后，系统按顺序或随机生成该分类下的题目供用户依次作答。

**业务规则：**

- 已做过的题目标记状态（已做/未做/做错）
- 用户可选择仅刷未做过的题或全部题
- 支持在练习过程中随时退出

#### 4.3.5 每日推荐

**功能描述：** 系统每天定时（如凌晨00:00）通过推荐算法为每位用户生成5道个性化推荐题目。

**业务规则：**

- 每日推荐固定5道题目
- 推荐题目不重复（不推荐用户近期已做且已掌握的题目）
- 新用户基于注册时填写的兴趣方向和技术水平进行冷启动推荐
- 5道题中兼顾不同分类和难度
- 用户可在"每日推荐"页面查看今日推荐题目及完成进度
- 定时任务每日凌晨执行，生成结果存入推荐结果表

#### 4.3.6 错题回顾

**功能描述：** 收录用户答错或未掌握的题目，支持重新作答。

**业务规则：**

- 简答题：AI评分低于60分的题目自动收入错题本
- 代码题：未通过所有测试用例的题目自动收入错题本
- 判定依据为用户**最近一次**提交结果
- 用户可手动将题目从错题本中移除（标记为已掌握）
- 重新作答并达标后自动从错题本移除
- 错题列表支持按分类、难度筛选

### 4.4 推荐系统模块

#### 4.4.1 推荐数据采集

**显式数据：**

| 数据项 | 来源 | 用途 |
|--------|------|------|
| 兴趣方向 | 注册时填写 | 冷启动、基于内容推荐 |
| 技术水平 | 注册时填写 | 难度匹配 |
| 题目评分/反馈 | 答题后用户反馈 | 协同过滤评分矩阵 |

**隐式数据：**

| 数据项 | 采集方式 | 用途 |
|--------|----------|------|
| 刷题记录（含重复练习） | 自动记录 | 用户行为矩阵，多次做错加权 |
| 答题正确/错误 | 自动记录 | 用户能力评估 |
| 答题耗时 | 前端计时 | 题目难度评估、用户熟练度 |
| 题目收藏 | 用户主动操作 | 兴趣偏好 |
| 各分类正确率 | 自动统计 | 薄弱点识别 |
| 题目浏览但未作答 | 自动记录 | 兴趣/犹豫识别 |
| 页面停留时长 | 前端定期上报 | 学习投入度 |
| 评论行为 | 自动记录 | 参与度评估 |

#### 4.4.2 推荐算法

**算法一：基于用户的协同过滤（User-based CF）**

- 核心思路：找到与当前用户刷题行为相似的其他用户，将相似用户做过但当前用户未做的题目推荐给当前用户
- 相似度度量：Pearson相关系数 或 余弦相似度
- 构建用户-题目评分矩阵（评分可结合正确性、耗时、反馈等综合计算）
- 取Top-K相似用户的高分题目作为推荐候选
- 相似度结果持久化存储于`user_similarities`表

**算法二：基于物品的协同过滤（Item-based CF）**

- 核心思路：计算题目之间的相似度，根据用户已做题目推荐与之相似的其他题目
- 相似度计算：基于所有用户对两道题的评分向量的余弦相似度
- 适合题目数量相对稳定的场景（本系统题库由管理员手动维护，更新频率低）
- 相似度结果持久化存储于`question_similarities`表

**算法三：基于内容的推荐（Content-based）**

- 核心思路：根据题目本身的属性（分类、难度、标签、关键词）与用户偏好进行匹配
- 构建题目特征向量（TF-IDF或基于标签的向量）
- 构建用户偏好向量（基于历史做题的题目特征加权平均）
- 计算用户偏好与候选题目的相似度进行排序

**混合策略：**

- 可对以上三种算法的推荐结果按权重进行线性融合
- 针对冷启动用户（做题数少）更多依赖基于内容的推荐
- 针对活跃用户更多依赖协同过滤

#### 4.4.3 相似度持久化存储

采用"定时批量刷新"策略：在每日定时任务中先计算相似度矩阵并写入数据库，再基于相似度表生成推荐结果。

**存储规则：**

- 每个用户只存最相似的Top-K（如前20个）用户
- 每道题目只存最相似的Top-K题目
- `user_a_id < user_b_id`约束去重，避免重复存储
- 每次定时任务全量刷新（删除旧数据再插入）
- `algorithm`字段支持同时存储多种算法的结果，服务于管理员对比面板

#### 4.4.4 推荐算法效果对比（管理员功能）

**功能描述：** 管理员可在后台对比面板中切换不同推荐算法，查看推荐效果差异。

**对比维度与指标：**

| 指标 | 说明 |
|------|------|
| 准确率（Precision） | 推荐的题目中用户实际做了的比例 |
| 召回率（Recall） | 用户做的题目中由推荐命中的比例 |
| F1-Score | 准确率和召回率的调和平均 |
| NDCG | 排序质量评估 |
| 覆盖率（Coverage） | 推荐能覆盖多少题目 |
| 多样性（Diversity） | 推荐列表中题目的多样性 |

**展示方式：**

- 可选择对比的算法组合
- 以图表形式（柱状图、折线图）直观展示各指标对比
- 支持按时间段筛选数据
- 查看各算法的推荐题目样例

## 5. 非功能性需求

### 5.1 性能需求

| 指标 | 要求 |
|------|------|
| 并发用户数 | 支持100人同时在线 |
| 页面响应时间 | 普通页面 < 2秒 |

| 推荐计算 | 每日推荐定时离线计算，无实时性要求 |

### 5.2 安全需求

- 用户密码使用Django内置的PBKDF2算法加密存储
- 代码题判题在Docker沙箱中隔离执行，防止恶意代码
- API接口需身份认证（JWT/Session）
- 防止SQL注入、XSS攻击（Django ORM和Vue.js自带防护）
- 邮箱验证码防刷机制（频率限制）

### 5.3 可用性需求

- 界面设计简洁清晰，符合程序员使用习惯
- 代码编辑器提供语法高亮、基本自动补全
- 响应式布局适配主流桌面浏览器（Chrome、Firefox、Edge）
- 系统操作流畅，核心路径不超过3次点击

### 5.4 可维护性需求

- 前后端分离架构，降低耦合
- 数据库设计支持从SQLite平滑迁移至MySQL
- 推荐算法模块化设计，方便新增或替换算法
- 代码规范，关键模块有注释

### 5.5 数据需求

- 题库初始规模：几百道（覆盖主流岗位方向）
- 数据备份：SQLite文件定期备份
- 推荐算法训练数据：通过模拟数据生成脚本产生（详见第11章）

### 6.2 冷启动策略

- **新用户冷启动：** 基于注册时选择的兴趣方向和技术水平，使用基于内容的推荐推送匹配题目
- **新题目冷启动：** 新导入的题目根据其分类和难度属性推荐给偏好匹配的用户
- **随着用户行为积累，逐步增加协同过滤的权重**

### 6.3 定时任务流程

每日凌晨执行（Django Celery / crontab）：

1. 计算用户相似度矩阵，写入`user_similarities`表
2. 计算题目相似度矩阵，写入`question_similarities`表
3. 基于相似度表，运行各推荐算法生成推荐候选
4. 混合策略融合，为每位活跃用户生成5道每日推荐
5. 存入`daily_recommendations`表
6. 计算各算法评估指标，存入评估结果表

## 11. 测试数据生成策略

由于系统为演示级原型，上线初期缺少真实用户行为数据，需要通过模拟数据生成脚本来支撑推荐算法的训练和效果展示。

### 11.1 模拟数据规模

| 数据类型 | 规模 | 说明 |
|----------|------|------|
| 模拟用户 | 100个 | 覆盖不同兴趣方向、技术水平、学历的组合 |
| 题库 | 几百道（真实题目） | 手动收集整理 |
| 模拟答题记录 | 每用户50-200条 | 总计约5000-20000条记录 |
| 模拟收藏 | 每用户5-30条 | 随机收藏 |
| 模拟评分反馈 | 每用户10-50条 | 随机评分 |

### 11.2 模拟策略

- **用户画像模拟：** 为每个模拟用户随机分配兴趣方向和技术水平，确保分布合理
- **行为模式模拟：** 模拟用户倾向于做自己感兴趣方向的题目（80%概率），偶尔做其他方向（20%概率），体现行为的"聚集性"，这对协同过滤至关重要
- **正确率模拟：** 技术水平高的用户在简单题上正确率高，在困难题上正确率适中；技术水平低的用户反之。加入随机噪声模拟真实情况
- **重复练习模拟：** 部分题目模拟多次作答，分数逐次提升，模拟学习效果

### 11.3 生成脚本

编写Django Management Command，支持一键生成全部模拟数据：

```bash
python manage.py generate_mock_data --users 100 --min-submissions 50 --max-submissions 200
```

脚本执行后自动填充`users`、`submission_records`、`user_question_status`、`collections`、`user_question_ratings`等表，并触发一次推荐算法计算。
